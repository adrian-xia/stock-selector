## 0. 文档目的

本文档用于指导在现有 `stock-selector` 上落地"盘后自动投研与交易计划系统（StarMap）"，形成可执行、可回放、可监控的完整闭环。  
本版为最终实施基线（V5），整合了两轮外部架构 Review：
- 新闻源可用性 PoC 与数据来源边界
- V1/V2 能力分层（先可用再增强）
- 多因子量纲统一与小样本畸变防护
- Tushare 就绪探针的固定下限与冷启动策略
- plan_status 的可实施边界与盘中状态延后
- 新闻去重与 LLM Token 截断防爆策略
- Orchestrator 统一降级标记与状态清理

## 1. 背景与问题定义

### 1.1 当前系统能力（已具备）

现有系统已具备完整的盘后链路和策略引擎：
- 调度链路：`app/scheduler/jobs.py` 的 `run_post_market_chain`
- 策略执行：`app/strategy/pipeline.py` + `StrategyFactory`
- 数据底座：`DataManager` + PostgreSQL + Redis
- 报告通知：`app/scheduler/report.py` + `NotificationManager`
- AI 基础设施：`app/ai/clients/gemini.py` + `app/ai/manager.py` + `app/ai/news_analyzer.py`

### 1.2 当前痛点

1. 策略信号与"宏观/行业叙事"没有结构化联动，导致信号解释弱。  
2. 盘后报告偏"结果展示"，缺少"次日可执行的交易计划模板"。  
3. 风险控制主要在策略参数层，缺少市场状态的统一上层仓位约束。  
4. 新闻与非结构化信息没有形成可回溯的结构化资产（落表+评分）。

### 1.3 目标

构建"宏观 -> 市场 -> 行业 -> 个股 -> 交易计划"的漏斗式决策层：
- 对外输出可执行计划（Entry/Stop/TP/Position）
- 对内保留结构化中间结果，支持复盘与回测
- 与现有策略框架"增量集成"，不推翻现有架构

---

## 2. 设计原则

1. **增量改造，不破坏现有主链路**：StarMap 作为盘后链路增强阶段。  
2. **结构化优先**：LLM 输出必须通过 JSON Schema 校验后才可入库。  
3. **降级可用**：LLM/新闻失败自动降级为纯量化模式。  
4. **可追溯**：每个计划都能追踪到策略信号、行业共振与宏观依据。  
5. **执行纪律先于叙事**：止损规则硬约束，不被叙事覆盖。  
6. **词表对齐优先**：行业标签必须可映射到标准 `sector_code`。  
7. **特征开关可回滚**：所有新增能力有 feature flag 控制。  
8. **V1先稳再强**：V1 不引入高运维复杂度组件（如 embedding 对齐链路）。

## 3. 总体架构

### 3.1 逻辑分层

1. 宏观事件层（News + LLM）  
2. 市场状态层（宽基 + 情绪 + 风险）  
3. 行业共振层（消息/资金/趋势融合）  
4. 个股执行层（策略结果二次编排）  
5. 计划交付层（交易计划 + 报告 + 通知）

### 3.2 架构图（ASCII）

```text
               ┌──────────────────────────────────┐
               │ run_post_market_chain(target)    │
               └──────────────────────────────────┘
                            │
                            ▼
         ┌───────────────────────────────┐
         │ Existing Pipeline (L1~L5)     │
         │ strategy_picks / trade_plan   │
         └───────────────────────────────┘
                            │
                            ▼
 ┌──────────────────────────────────────────────────────────┐
 │ StarMap Orchestrator (new)                              │
 │ 0) Expire stale plans (PENDING→EXPIRED)                │
 │ 1) Readiness Probe                                      │
 │ 2) News Ingest → 3) LLM Struct (Schema + Align)       │
 │ 4) Market Score → 5) Sector Resonance                 │
 │ 6) Plan Fusion → 7) Report                            │
 │ ── degrade_flags 贯穿全链路 ──                         │
 └──────────────────────────────────────────────────────────┘
      │                │                  │
      ▼                ▼                  ▼
 macro_signal_daily  sector_resonance   trade_plan_daily_ext
      │                                   │
      └──────────────► post_market_report ◄───────────────┘
                            │
                            ▼
                      Telegram / API
```

### 3.3 Orchestrator 统一降级标记

Orchestrator 在执行 6 个子步骤时，维护一个 `degrade_flags: list[str]` 列表：
- 每个模块失败/降级时追加标记（如 `"NEWS_FETCH_FAILED"`, `"LLM_DEGRADED"`, `"SECTOR_SKIPPED"`）
- 最终传递给报告模块，在报告末尾透明展示"今日降级项"
- 同时写入 `task_log` 供运维巡检

## 6. 核心算法与规则

### 6.1 市场状态评分（Market Regime）

输入：
- 指数：上证、深成、沪深300、创业板
- 情绪：涨停数、跌停数、连板高度、昨日涨停溢价
- 量能：成交额相对近 20 日均值

打分公式：

```text
# 约定：risk_score 越高表示风险越大（0=低风险，100=高风险）
risk_score = 0.35*volatility_risk + 0.25*trend_risk + 0.20*breadth_risk + 0.20*sentiment_risk

position_cap:
  risk_score >= 70 -> 0.30
  40 <= risk_score < 70 -> 0.60
  risk_score < 40 -> 0.90
```

子项分段映射（V1，可解释且可直接编码）：

**trend_risk**（基于指数收盘与 MA60 偏离度）：

| `close_vs_ma60` | risk 值 |
|------------------|---------|
| ≤ -3% | 80 |
| -3% ~ 0% | 60 |
| 0% ~ +3% | 40 |
| > +3% | 20 |

补充：同时参考 `close_vs_ma20`，若 close < MA20 则 risk += 10（加罚项，上限 90）。

**breadth_risk**（`advance_decline_ratio = 上涨家数 / 下跌家数`）：

| `advance_decline_ratio` | risk 值 |
|--------------------------|---------|
| < 0.8 | 80 |
| 0.8 ~ 1.0 | 60 |
| 1.0 ~ 1.2 | 40 |
| > 1.2 | 20 |

**sentiment_risk**（基于涨跌停结构量化）：

| 条件 | risk 值 | 说明 |
|------|---------|------|
| `limit_up / limit_down < 0.5` | 80 | 跌停远多于涨停 = 冰点 |
| `0.5 ≤ ratio < 1.0` | 60 | 偏弱 |
| `1.0 ≤ ratio < 2.0` | 40 | 中性 |
| `ratio ≥ 2.0` | 20 | 强势 |

辅助修正：连板高度 ≤ 2 时 sentiment_risk += 10；昨日涨停溢价 < -3% 时 sentiment_risk += 10。

**volatility_risk**（`index_atr_pct = ATR(14) / close * 100`）：

| `index_atr_pct` | risk 值 |
|------------------|---------|
| > 3.5% | 80 |
| 2.5% ~ 3.5% | 60 |
| 1.5% ~ 2.5% | 40 |
| < 1.5% | 20 |

输出：
- `market_regime`: bull / range / bear
- `risk_score`: 0~100（高分=高风险）
- `position_cap`: 建议总仓位上限

注：V1 先使用分段映射保证可解释性，M4 再做参数校准。

### 6.2 行业共振评分

```text
# V1 初始权重（先降低 LLM 噪声影响）
final_score = 0.20*news_score + 0.45*moneyflow_score + 0.35*trend_score
confidence = min(100, 0.6*data_completeness + 0.4*signal_consistency)
```

分数口径：
- `news_score`: 0~100，来自结构化事件强度（非原始 LLM 概率）
- `moneyflow_score`: 0~100，行业资金因子归一化分
- `trend_score`: 0~100，行业趋势因子归一化分

约束：
- 必须满足至少两项 > 60 分才进入 focus list
- 若 `negative_sectors` 命中，则 `news_score` 降权
- M4 必须执行权重校准回测，V1 权重仅为启动参数

### 6.3 个股计划融合（含量纲统一）

计划候选来源：
1. `strategy_picks`（已有）
2. 行业共振 TopN 成分股（可选）

先做横截面归一化，再做加权：

```text
strategy_score_n   = percentile_rank(strategy_score)
sector_score_n     = percentile_rank(sector_score)
moneyflow_score_n  = percentile_rank(moneyflow_raw)
liquidity_score_n  = percentile_rank(turnover_ratio)

stock_rank = 0.50*strategy_score_n + 0.25*sector_score_n
           + 0.15*moneyflow_score_n + 0.10*liquidity_score_n
```

归一化范围约束（避免小样本畸变）：
- `strategy_score_n/moneyflow_score_n/liquidity_score_n` 必须先在全市场基座计算
- 行业内候选过滤只能发生在归一化之后
- 当候选池样本数 `< 30` 时，禁止在小样本内重新做 percentile
- 小样本替代方案：使用全市场 percentile 列直接过滤；若缺失则退化为 min-max 缩放

仓位建议：

```text
position_suggestion = base_position * (1 - market_risk_score/100) * confidence_factor
```

### 6.4 极端风险兜底规则

当次日开盘出现跳空穿透止损位：
- 若 `open_price <= stop_loss * (1 - gap_threshold)`，触发 `emergency_exit_rule`
- 默认规则：`开盘竞价/开盘首分钟风控离场`
- 同时打上 `risk_flags += ["GAP_DOWN_BREACH"]`

结构化规则（存入 `emergency_exit_config`，V1 即可落地）：

```json
{
  "trigger": {
    "type": "gap_down_breach",
    "threshold": 0.02
  },
  "action": {
    "type": "exit_at_open",
    "max_slippage_bps": 50
  },
  "priority": 1
}
```

说明：
- `emergency_exit_text` 仅用于报告展示（人工提醒，V1 不自动执行）
- `emergency_exit_config` 供规则引擎解析（为 V2 盘中自动风控预留）

## 7. LLM 设计（Prompt + Schema + 对齐 + 容错）

### 7.1 行业实体对齐（V1/V2 分层）

V1（必须）：
1. **受控词表约束**：Prompt 注入标准行业枚举（来自 Tushare 板块字典）。
2. **别名映射层**：维护 alias（如"飞行汽车"->"低空经济"）。
3. **硬降级**：对齐失败率超过阈值（建议 30%）则降级纯量化。

V2（可选增强）：
4. **语义兜底层**：embedding 相似度映射（引入额外运维复杂度后再开启）。

**前置准备**：M1 阶段需从 Tushare 板块字典导出标准 `sector_code → sector_name` 映射表，并整理高频 alias 词典，否则 `aligner.py` 无字典可查。

### 7.2 Prompt 约束

- 强制输出 JSON
- 禁止 markdown 包裹
- 必须包含行业标签数组
- 摘要长度限制（<=200 汉字，推荐条目式输出）
- 若无法判定行业，必须输出空数组 `[]`，不得虚构

### 7.3 Schema 校验与自适应重试

- 使用 `pydantic` 严格校验类型
- 第一次失败：返回校验错误并重试
- 第二次重试：追加容错指令（允许空值、禁止捏造）
- 两次失败后：降级 `risk_appetite=mid`，行业空数组，进入纯量化模式

### 7.4 模型与成本控制

- 默认模型可配置（复用 `app.ai.manager` + `app/ai/clients` 体系）
- 单次最大 token 与超时控制
- 记录 `model_name/prompt_version/content_hash` 便于回放

**Token 截断防爆策略**（Review 补充）：
- 在 `prompts.py` 拼接新闻 payload 时，按时间倒序排列
- 强制截断到模型最大上下文的 **80% 长度**，避免"中间注意力丢失（Lost in the middle）"
- 截断后在 prompt 结尾追加 `[System: 以上为今日核心财经资讯，请据此输出JSON]`，防止模型被截断的半截话输出崩溃

### 7.5 新闻去重策略（Review 补充）

同一条宏观事件（如央行降准）可能在多个新闻源重复出现，需在 LLM 调用前去重：

- V1 方案（推荐）：在 `dedupe.py` 中使用 **Jaccard 分词相似度**
  - 对标题分词后计算词交集占比
  - 阈值 0.7：相似度 ≥ 0.7 的新闻视为重复，保留时间最早的一条
  - 可解决 ~80% 的通稿重复问题
- V2 可选：MinHash 或 SimHash（新闻量大时升级）

## 9. 调度编排与时序

### 9.1 数据就绪探针

在 StarMap 阶段前执行 `readiness probe`：
- 检查 `daily/daily_basic/moneyflow` 当日覆盖率是否达到阈值
- 核心就绪判定：`today_count >= max(avg_count_20d * 0.95, floor_count)`（按字段分别计算）
- `floor_count` 建议初始值：4000（可配置）
- 冷启动（不足 20 个交易日）时优先使用 `prev_trade_count` 与 `floor_count`
- 不满足则延迟重试（最多 N 次），仍失败则降级模式

### 9.2 日内时序（建议）

- 15:30：行情数据同步（现有）
- 16:00：数据就绪探针
- 16:05：清理过期计划（PENDING→EXPIRED）
- 16:10：新闻抓取与清洗（含去重）
- 16:20：LLM 结构化提取
- 16:30：市场评分与行业共振
- 16:40：交易计划生成
- 16:45：报告生成与通知

### 9.3 幂等与并发

- 以 `trade_date` 作为幂等键
- 所有落表使用 UPSERT
- 对 `macro_signal_daily`，`ON CONFLICT (trade_date) DO UPDATE` 必须覆盖 `content_hash = EXCLUDED.content_hash`（以及 `raw_payload/model_name/prompt_version/updated_at`）
- 调度锁沿用现有 `sync_lock`

### 9.4 重试策略

- 新闻抓取失败：重试 3 次，指数退避
- LLM 失败：最多 2 次（自适应提示），失败降级
- 落表失败：单模块失败不阻断主链路，记录告警

### 9.5 新闻源 PoC（实施前置）

M1 开始前必须完成新闻源 PoC，并形成可用性结论：
- 数据来源类型（官方 API / 聚合 API / 合规爬取）
- 稳定性（交易日覆盖率、延迟、失败率）
- 字段质量（标题、正文、时间戳、来源）
- 法务与调用限制（频率、版权、条款）

PoC 未通过时，StarMap 仅启用"纯量化 + 公告情感"最小模式。

同时约束：
- `starmap_news_source_whitelist` 为空时，新闻模块默认 `skip`
- 系统自动降级到纯量化路径，不视为失败中断

### 9.6 计划状态清理（Review 补充）

每日 StarMap Orchestrator 执行的第一步为清理过期计划：

```sql
UPDATE trade_plan_daily_ext
SET plan_status = 'EXPIRED', updated_at = NOW()
WHERE plan_status = 'PENDING'
  AND trade_date < :today;
```

集成在 Orchestrator 入口，而非独立定时器，避免增加调度复杂度。

## 10. 失败降级矩阵

| 模块 | 失败影响 | 降级动作 | 是否阻断盘后链路 |
|------|----------|----------|------------------|
| 数据就绪探针 | 数据不完整 | 延迟重试，失败后降级纯量化 | 否 |
| 新闻抓取 | 无宏观输入 | 使用前一交易日宏观信号或空值 | 否 |
| LLM 提取 | 无语义标签 | 进入纯量化模式 | 否 |
| 市场评分 | 无仓位约束 | 使用默认仓位上限 0.5 | 否 |
| 行业共振 | 无行业加权 | 使用策略原始排序 | 否 |
| 计划生成 | 无增强计划 | 保留原 strategy_picks 报告 | 否 |
| 报告推送 | 用户收不到 | 写 task_log + 重试推送 | 否 |

## 12. 安全与风控

1. LLM 输出不直接触发下单，仅供计划建议。  
2. 所有交易计划必须包含止损字段，缺失则拒绝发布。

4. 报告中明确"非投资建议，策略需纪律执行"。  
5. 当触发极端缺口规则时，优先执行 `emergency_exit_rule`。

## 16. 关键风险与缓解

| 风险 | 影响 | 缓解 |
|------|------|------|
| 新闻源 PoC 失败 | M1 缩减 50%，无宏观层 | 降级为"公告情感 + 纯量化"模式，市场评分/行业共振仍可工作 |
| Tushare moneyflow 限频 | 行业共振 moneyflow_score 缺失 | 缓存 + 单日批量调用；缺失时 moneyflow_score=50（中性） |
| LLM 行业对齐准确率低 | 降级触发频繁 | 30% 阈值可配置调节；逐步扩充 alias 词典 |
| WBS 超时 | 总工期变长 | Phase 0 PoC 结论决定 M1 scope，scope 可收可放 |
| LLM "lost in the middle" | 宏观分析漏掉关键新闻 | Token 截断到 80% + 按时间倒序排列 |
| 新闻通稿重复 | LLM 过度放大单一事件权重 | Jaccard 去重阈值 0.7 过滤 |

## 17. 与"高位回落企稳二次启动"策略的衔接

针对 `peak-pullback-stabilization`：

1. 作为 `source_strategy` 之一进入计划融合层。  
2. 当所属行业进入 `focus_sectors` 且 `sector_score>70`，排序加权。  
3. 当 `market_risk_score>70` 时自动降仓，即便形态命中也不满仓。  
4. 报告输出"形态信号 + 行业共振 + 宏观风险"三段式解释。  
5. 次日若跳空穿透止损，执行 `emergency_exit_rule`。

## 20. 代码实现防坑清单

1. 横截面百分位统一在全市场算完再过滤，避免行业小样本 rank 畸变。  
2. `trade_date` 幂等 UPSERT 时必须覆盖 `content_hash`，保证审计一致性。  
3. LLM 摘要限制放宽到 200 字，并要求条目式输出，减少信息丢失。  
4. Readiness Probe 使用 `max(20日均值*95%, floor_count)`，并处理冷启动。  
5. 新闻拼接 LLM 前必须先去重（Jaccard ≥ 0.7 去除通稿），再按时间倒序排列后截断到 80% Token 上下文。  
6. Orchestrator 入口第一步清理 PENDING→EXPIRED，避免前端查出历史堆积的无效计划。  
7. `sentiment_risk` 实现时使用 `涨停数/跌停数` 量化比值，不要用主观判定。  
8. `sector_code` 映射表必须在 M1 一并准备，否则 `aligner.py` 无字典可查。  
9. StarMap 所有 LLM 调用走 `app/ai/clients/gemini.py`，不要新建 client。  
10. 每个子步骤的降级标记追加到 `degrade_flags`，最终在报告末尾统一展示。

## 0. 文档目的

本文档用于指导在现有 `stock-selector` 上落地“盘后自动投研与交易计划系统（StarMap）”，形成可执行、可回放、可监控的完整闭环。  
本版在 V3 基础上合并了外部需求 Review，重点强化：
- 新闻源可用性 PoC 与数据来源边界
- V1/V2 能力分层（先可用再增强）
- 多因子量纲统一与小样本畸变防护
- Tushare 就绪探针的固定下限与冷启动策略
- plan_status 的可实施边界与盘中状态延后

## 1. 背景与问题定义

### 1.1 当前系统能力（已具备）

现有系统已具备完整的盘后链路和策略引擎：
- 调度链路：`app/scheduler/jobs.py` 的 `run_post_market_chain`
- 策略执行：`app/strategy/pipeline.py` + `StrategyFactory`
- 数据底座：`DataManager` + PostgreSQL + Redis
- 报告通知：`app/scheduler/report.py` + `NotificationManager`

### 1.2 当前痛点

1. 策略信号与“宏观/行业叙事”没有结构化联动，导致信号解释弱。  
2. 盘后报告偏“结果展示”，缺少“次日可执行的交易计划模板”。  
3. 风险控制主要在策略参数层，缺少市场状态的统一上层仓位约束。  
4. 新闻与非结构化信息没有形成可回溯的结构化资产（落表+评分）。

### 1.3 目标

构建“宏观 -> 市场 -> 行业 -> 个股 -> 交易计划”的漏斗式决策层：
- 对外输出可执行计划（Entry/Stop/TP/Position）
- 对内保留结构化中间结果，支持复盘与回测
- 与现有策略框架“增量集成”，不推翻现有架构

---

## 2. 设计原则

1. **增量改造，不破坏现有主链路**：StarMap 作为盘后链路增强阶段。  
2. **结构化优先**：LLM 输出必须通过 JSON Schema 校验后才可入库。  
3. **降级可用**：LLM/新闻失败自动降级为纯量化模式。  
4. **可追溯**：每个计划都能追踪到策略信号、行业共振与宏观依据。  
5. **执行纪律先于叙事**：止损规则硬约束，不被叙事覆盖。  
6. **词表对齐优先**：行业标签必须可映射到标准 `sector_code`。  
7. **特征开关可回滚**：所有新增能力有 feature flag 控制。  
8. **V1先稳再强**：V1 不引入高运维复杂度组件（如 embedding 对齐链路）。

## 5. 数据模型设计（详细）

### 5.1 表：`macro_signal_daily`

用途：存储每日宏观结构化信号（LLM 提取结果）。

```sql
CREATE TABLE IF NOT EXISTS macro_signal_daily (
  id BIGSERIAL PRIMARY KEY,
  trade_date DATE NOT NULL UNIQUE,
  risk_appetite VARCHAR(8) NOT NULL, -- high/mid/low
  global_risk_score NUMERIC(5,2) NOT NULL, -- 0~100
  positive_sectors JSONB NOT NULL DEFAULT '[]'::jsonb,
  negative_sectors JSONB NOT NULL DEFAULT '[]'::jsonb,
  macro_summary TEXT NOT NULL,
  key_drivers JSONB NOT NULL DEFAULT '[]'::jsonb,
  raw_payload JSONB NOT NULL,
  content_hash VARCHAR(64) NOT NULL, -- 新增: 输入新闻内容哈希
  model_name VARCHAR(128) NOT NULL,
  prompt_version VARCHAR(32) NOT NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_macro_signal_trade_date ON macro_signal_daily(trade_date DESC);
CREATE INDEX idx_macro_signal_hash ON macro_signal_daily(content_hash);
```

### 5.2 表：`sector_resonance_daily`

用途：存储每日行业共振评分。

```sql
CREATE TABLE IF NOT EXISTS sector_resonance_daily (
  id BIGSERIAL PRIMARY KEY,
```

## 6. 核心算法与规则

### 6.1 市场状态评分（Market Regime）

输入：
- 指数：上证、深成、沪深300、创业板
- 情绪：涨停数、跌停数、连板高度、昨日涨停溢价
- 量能：成交额相对近 20 日均值

示例打分：

```text
# 约定：risk_score 越高表示风险越大（0=低风险，100=高风险）
risk_score = 0.35*volatility_risk + 0.25*trend_risk + 0.20*breadth_risk + 0.20*sentiment_risk

trend_risk      = f(close_vs_ma20, close_vs_ma60)
breadth_risk    = f(advance_decline_ratio, up_down_median)
sentiment_risk  = f(limit_up_down_ratio, 连板高度, 昨日涨停溢价)
volatility_risk = f(index_atr_pct, intraday_range_pct)

position_cap:
  risk_score >= 70 -> 0.30
  40 <= risk_score < 70 -> 0.60
  risk_score < 40 -> 0.90
```

输出：
- `market_regime`: bull / range / bear
- `risk_score`: 0~100（高分=高风险）
- `position_cap`: 建议总仓位上限

### 6.2 行业共振评分

```text
# V1 初始权重（先降低 LLM 噪声影响）
final_score = 0.20*news_score + 0.45*moneyflow_score + 0.35*trend_score
confidence = min(100, 0.6*data_completeness + 0.4*signal_consistency)
```

分数口径：
- `news_score`: 0~100，来自结构化事件强度（非原始 LLM 概率）
- `moneyflow_score`: 0~100，行业资金因子归一化分
- `trend_score`: 0~100，行业趋势因子归一化分

约束：
- 必须满足至少两项 > 60 分才进入 focus list
- 若 `negative_sectors` 命中，则 `news_score` 降权
- M4 必须执行权重校准回测，V1 权重仅为启动参数

### 6.3 个股计划融合（含量纲统一）

计划候选来源：
1. `strategy_picks`（已有）
2. 行业共振 TopN 成分股（可选）

先做横截面归一化，再做加权：

```text
strategy_score_n   = percentile_rank(strategy_score)
sector_score_n     = percentile_rank(sector_score)
moneyflow_score_n  = percentile_rank(moneyflow_raw)
liquidity_score_n  = percentile_rank(turnover_ratio)

stock_rank = 0.50*strategy_score_n + 0.25*sector_score_n
           + 0.15*moneyflow_score_n + 0.10*liquidity_score_n
```

归一化范围约束（避免小样本畸变）：
- `strategy_score_n/moneyflow_score_n/liquidity_score_n` 必须先在全市场基座计算
- 行业内候选过滤只能发生在归一化之后
- 当候选池样本数 `< 30` 时，禁止在小样本内重新做 percentile
- 小样本替代方案：使用全市场 percentile 列直接过滤；若缺失则退化为 min-max 缩放

仓位建议：

```text
position_suggestion = base_position * (1 - market_risk_score/100) * confidence_factor
```

### 6.4 极端风险兜底规则（新增）

当次日开盘出现跳空穿透止损位：
- 若 `open_price <= stop_loss * (1 - gap_threshold)`，触发 `emergency_exit_rule`
- 默认规则：`开盘竞价/开盘首分钟风控离场`
- 同时打上 `risk_flags += ["GAP_DOWN_BREACH"]`

## 7. LLM 设计（Prompt + Schema + 对齐 + 容错）

### 7.1 行业实体对齐（V1/V2 分层）

V1（必须）：
1. **受控词表约束**：Prompt 注入标准行业枚举（来自 Tushare 板块字典）。
2. **别名映射层**：维护 alias（如“飞行汽车”->“低空经济”）。
3. **硬降级**：对齐失败率超过阈值（建议 30%）则降级纯量化。

V2（可选增强）：
4. **语义兜底层**：embedding 相似度映射（引入额外运维复杂度后再开启）。

### 7.2 Prompt 约束

- 强制输出 JSON
- 禁止 markdown 包裹
- 必须包含行业标签数组
- 摘要长度限制（<=200 汉字，推荐条目式输出）
- 若无法判定行业，必须输出空数组 `[]`，不得虚构

### 7.3 Schema 校验与自适应重试

- 使用 `pydantic` 严格校验类型
- 第一次失败：返回校验错误并重试
- 第二次重试：追加容错指令（允许空值、禁止捏造）
- 两次失败后：降级 `risk_appetite=mid`，行业空数组，进入纯量化模式

### 7.4 模型与成本控制

- 默认模型可配置（复用 `app.ai.manager` + `app/ai/clients` 体系）
- 单次最大 token 与超时控制
- 记录 `model_name/prompt_version/content_hash` 便于回放

## 9. 调度编排与时序

### 9.1 数据就绪探针（新增）

在 StarMap 阶段前执行 `readiness probe`：
- 检查 `daily/daily_basic/moneyflow` 当日覆盖率是否达到阈值
- 核心就绪判定：`today_count >= max(avg_count_20d * 0.95, floor_count)`（按字段分别计算）
- `floor_count` 建议初始值：4000（可配置）
- 冷启动（不足 20 个交易日）时优先使用 `prev_trade_count` 与 `floor_count`
- 不满足则延迟重试（最多 N 次），仍失败则降级模式

### 9.2 日内时序（建议）

- 15:30：行情数据同步（现有）
- 16:00：数据就绪探针
- 16:10：新闻抓取与清洗
- 16:20：LLM 结构化提取
- 16:30：市场评分与行业共振
- 16:40：交易计划生成
- 16:45：报告生成与通知

### 9.3 幂等与并发

- 以 `trade_date` 作为幂等键
- 所有落表使用 UPSERT
- 对 `macro_signal_daily`，`ON CONFLICT (trade_date) DO UPDATE` 必须覆盖 `content_hash = EXCLUDED.content_hash`（以及 `raw_payload/model_name/prompt_version/updated_at`）
- 调度锁沿用现有 `sync_lock`

### 9.4 重试策略

- 新闻抓取失败：重试 3 次，指数退避
- LLM 失败：最多 2 次（自适应提示），失败降级
- 落表失败：单模块失败不阻断主链路，记录告警

### 9.5 新闻源 PoC（实施前置）

M1 开始前必须完成新闻源 PoC，并形成可用性结论：
- 数据来源类型（官方 API / 聚合 API / 合规爬取）
- 稳定性（交易日覆盖率、延迟、失败率）
- 字段质量（标题、正文、时间戳、来源）
- 法务与调用限制（频率、版权、条款）

PoC 未通过时，StarMap 仅启用“纯量化 + 公告情感”最小模式。

## 10. 失败降级矩阵

| 模块 | 失败影响 | 降级动作 | 是否阻断盘后链路 |
|------|----------|----------|------------------|
| 数据就绪探针 | 数据不完整 | 延迟重试，失败后降级纯量化 | 否 |
| 新闻抓取 | 无宏观输入 | 使用前一交易日宏观信号或空值 | 否 |
| LLM 提取 | 无语义标签 | 进入纯量化模式 | 否 |
| 市场评分 | 无仓位约束 | 使用默认仓位上限 0.5 | 否 |
| 行业共振 | 无行业加权 | 使用策略原始排序 | 否 |
| 计划生成 | 无增强计划 | 保留原 strategy_picks 报告 | 否 |
| 报告推送 | 用户收不到 | 写 task_log + 重试推送 | 否 |

## 12. 安全与风控

1. LLM 输出不直接触发下单，仅供计划建议。  
2. 所有交易计划必须包含止损字段，缺失则拒绝发布。  
3. 对高波动标的设置单票仓位上限（如 15%）。  
4. 报告中明确“非投资建议，策略需纪律执行”。  
5. 当触发极端缺口规则时，优先执行 `emergency_exit_rule`。

## 14. 测试策略

### 14.1 单元测试

- Prompt schema 校验
- 行业实体对齐测试（V1: 词表/alias/硬降级；V2: embedding）
- 市场评分函数边界测试
- 归一化函数一致性测试
- 极端缺口规则测试

### 14.2 集成测试

- 模拟完整盘后链路（含降级路径）
- 数据落表与 API 查询一致性

### 14.3 回放测试

- 选取 20 个交易日历史新闻回放
- 对比“仅策略” vs “策略+StarMap” 的候选排序稳定性

### 14.4 验收阈值（建议）

- 连续 10 个交易日稳定出报告
- LLM 结构化成功率 >= 90%
- 无 silent failure
- 关键字段（plan_status/content_hash）覆盖率 100%

### 14.5 成本评估（新增）

- Tushare 调用成本：关注 `moneyflow` 等高频接口的积分消耗和限频。
- LLM 调用成本：按“每日新闻条数 * 平均 token”估算，建议设置日预算与硬限流。
- V2 embedding 成本：若启用语义兜底，需额外评估向量生成与检索成本。

## 19. 合并后复审结论（本轮）

本版已吸收最新需求 Review 的核心问题：
- 明确了新闻源 PoC 是 M1 前置条件

## 20. 代码实现防坑清单（Review 补充）
